Step-by-Step Learning Guide — Tokenizer에서 Transformer 학습까지

Chapter 1 Foundation

Tokenizer: 텍스트를 숫자로 바꾸는 첫 번째 관문

컴퓨터는 문자열을 직접 이해하지 못해요. 신경망이 텍스트를 처리하려면 먼저 텍스트를 숫자의 나열로 변환해야 해요. 이 변환을 담당하는 것이 tokenizer에요. Tokenizer의 설계가 모델의 성능, 크기, 학습 효율에 직접적인 영향을 미쳐요.

▶ 1

Tokenizer란 무엇인가

text → numbers

Tokenizer는 텍스트 문자열을 token이라 부르는 단위로 쪼개고, 각 token에 고유한 정수 ID를 부여하는 도구에요. 예를 들어 "Hello world"라는 문자열이 tokenizer를 통과하면 [15496, 995] 같은 정수 배열이 돼요.

반대 과정도 가능해요. 정수 배열을 다시 텍스트로 복원하는 것을 decoding이라 불러요. Token은 단어일 수도 있고, 단어의 일부일 수도 있고, 심지어 단일 문자나 byte일 수도 있어요. 어떤 단위로 쪼개느냐가 tokenizer의 핵심 설계 결정이에요.

▶ 2

왜 단어 단위로 쪼개지 않나요?

vocab size 문제

가장 직관적인 방법은 공백 기준으로 단어를 나누는 것이에요. 하지만 이 방식에는 심각한 문제가 있어요.

영어만 해도 수십만 개의 단어가 있고, 고유명사, 전문용어, 신조어를 포함하면 사실상 무한해요. Vocabulary가 100만 개라면, 모델은 100만 x dim 크기의 embedding table을 가져야 해요. 이 프로젝트처럼 16MB 제한이 있으면 모델 대부분이 embedding table로 채워져서 실제 학습에 쓸 parameter가 남지 않아요.

반대로 개별 문자나 byte 단위로 쪼개면 vocabulary는 아주 작아지지만, sequence가 매우 길어져요. "Hello"가 5개의 token이 되면 모델이 처리해야 할 sequence 길이가 급격히 늘어나고, attention 연산량이 sequence 길이의 제곱에 비례하므로 비용이 폭증해요.

핵심 trade-off: vocabulary 크기가 크면 embedding table이 커지고, 작으면 sequence가 길어져요. 이 균형점을 찾는 것이 tokenizer 설계의 본질이에요.
▶ 3

BPE: Byte Pair Encoding

현대 tokenizer의 핵심 알고리즘

BPE: Byte Pair Encoding은 이 trade-off를 해결하는 가장 널리 쓰이는 알고리즘이에요. GPT, LLaMA, 이 프로젝트 모두 BPE 기반 tokenizer를 사용해요.

BPE의 원리는 단순해요. 먼저 텍스트를 가장 작은 단위인 개별 byte나 문자로 분해해요. 그 다음, 훈련 데이터에서 가장 자주 연속으로 등장하는 쌍을 반복적으로 합쳐서 새로운 token을 만들어요.

글자 블록으로 비유할 수 있어요. 처음에는 알파벳 한 글자씩 쓰인 블록만 있어요. "t"와 "h"가 자주 나란히 놓이면, 이 둘을 합쳐서 "th" 블록을 새로 만들어요. "th"와 "e"가 자주 붙으면 "the" 블록도 만들어요. 이 과정을 원하는 vocabulary 크기가 될 때까지 반복해요.

# 초기 상태: 모든 텍스트를 byte 단위로 분해 corpus = "the cat sat on the mat" tokens = ['t', 'h', 'e', ' ', 'c', 'a', 't', ...] # Step 1: 가장 빈번한 쌍 찾기 → ('t', 'h') 가 가장 많음 # → 새 token 'th' 추가, vocab = [..., 'th'] # Step 2: 이제 ('th', 'e') 가 가장 빈번 # → 새 token 'the' 추가, vocab = [..., 'th', 'the'] # Step 3: (' ', 'the') 가 빈번 # → 새 token ' the' 추가 # 원하는 vocab size에 도달할 때까지 반복

결과적으로 자주 등장하는 단어나 단어 조각은 하나의 token이 되고, 드문 단어는 여러 token으로 분해돼요. 이렇게 하면 vocabulary 크기를 제한하면서도 흔한 패턴은 효율적으로 표현할 수 있어요.

▶ 4

SentencePiece: BPE의 실용적 구현

이 프로젝트의 tokenizer library

SentencePiece는 Google이 만든 tokenizer library로, BPE 알고리즘을 실제로 구현한 도구에요. 이 프로젝트에서는 import sentencepiece as spm으로 사용해요.

SentencePiece의 특징은 텍스트를 공백 기준으로 미리 나누지 않는다는 것이에요. 대신 공백도 일반 문자처럼 취급하되, 단어 시작 위치에 특수 기호 ▁를 붙여요. 예를 들어 "Hello world"는 ["▁Hello", "▁world"]로 tokenize돼요. 이 ▁ 기호는 "이 token 앞에 공백이 있었다"는 의미에요.

Special tokens: SentencePiece는 일반 텍스트 token 외에도 control token을 지원해요. BOS: Beginning of Sequence, EOS: End of Sequence, UNK: Unknown 같은 특수 token이 있어요. 이 프로젝트에서 BOS token은 문서 경계를 표시하는 데 사용돼요.

Check Your Understanding

BPE tokenizer의 vocabulary 크기를 늘리면 어떤 변화가 생길까요?

Vocab 크기가 커지면 더 긴 단어나 구문이 하나의 token으로 표현되므로 sequence가 짧아져요. 하지만 embedding table의 행 수가 늘어나므로 table 자체는 커져요. 이 프로젝트에서 vocab=1024로 매우 작게 설정한 이유가 바로 16MB 크기 제한 때문이에요.
Chapter 2 Project-Specific

이 프로젝트의 Tokenizer: vocab_size = 1024

이 프로젝트는 vocab_size를 1024로 설정했어요. GPT-4의 100,000+이나 LLaMA의 32,000과 비교하면 극단적으로 작은 값이에요. 이 선택이 전체 모델 설계에 어떤 영향을 미치는지 살펴볼게요.

▶ 1

왜 1024인가: 16MB 제약의 산물

size budget 계산

Embedding table의 크기는 vocab_size x model_dim이에요. 이 프로젝트에서는 1024 x 512 = 524,288 parameter에요. 만약 vocab_size가 32,000이었다면 32,000 x 512 = 16,384,000 parameter가 돼요. fp32 기준으로 약 62MB, int8로 양자화해도 약 15.6MB에요.

Vocab SizeEmbedding Paramsint8 크기남은 budget
1,024524,288~0.5MB~15.5MB
8,1924,194,304~4MB~12MB
32,00016,384,000~15.6MB~0.4MB
Trade-off: vocab이 작으면 같은 텍스트를 표현하는 데 더 많은 token이 필요해요. "transformer"라는 단어가 vocab=100K에서는 1 token이지만, vocab=1024에서는 여러 token으로 쪼개져요. 이 프로젝트에서는 seq_len=1024로 고정해서 관리해요.
▶ 2

Tied Embedding: 하나의 행렬로 두 가지 역할

input & output 공유

보통 language model에는 두 개의 큰 행렬이 있어요. 하나는 token ID를 vector로 바꾸는 input embedding이고, 다른 하나는 vector를 다시 vocab 크기의 확률 분포로 바꾸는 output head에요. Tied embedding은 이 두 행렬을 동일한 weight로 공유하는 기법이에요.

이렇게 하면 parameter 수를 절반으로 줄이면서도, input과 output의 token 표현이 일관되도록 강제해요.

▶ 3

bits-per-byte: tokenizer에 독립적인 평가 지표

공정한 비교의 핵심

다른 vocab 크기의 모델을 어떻게 공정하게 비교할까요? 이 프로젝트는 bits-per-byte를 사용해요. 모델이 텍스트를 얼마나 잘 "압축"하는지를 원본 텍스트의 byte당 필요한 bit 수로 측정하는 것이에요.

# bits-per-byte 계산 과정 loss_in_nats = 2.35 bits_per_token = loss_in_nats / ln(2) # = 3.39 tokens_per_byte = total_tokens / total_bytes bits_per_byte = bits_per_token * tokens_per_byte

Tokenizer가 텍스트를 숫자로 변환한 결과물이 binary shard 파일이에요. 학습 시에는 이 파일들을 효율적으로 읽어서 모델에 공급해야 해요. 이 챕터에서는 데이터가 디스크에서 GPU까지 어떻게 흘러가는지 추적해요.

▶ 2

TokenStream: 끝없는 데이터 흐름

sequential read + auto-wrap

여러 권의 책이 선반에 꽂혀 있다고 상상해 보세요. TokenStream은 첫 번째 책부터 읽기 시작해서 끝까지 읽으면 자동으로 다음 책으로 넘어가고, 마지막 책까지 다 읽으면 다시 첫 번째 책으로 돌아가서 끝없이 읽어나가는 독서 로봇이에요.

class TokenStream: def take(self, n): chunks = [] remaining = n while remaining > 0: avail = self.tokens.numel() - self.pos if avail <= 0: self._advance_file() # 다음 shard continue k = min(remaining, avail) chunks.append(self.tokens[self.pos : self.pos + k]) self.pos += k remaining -= k return torch.cat(chunks)

이 설계는 deterministic해요. random shuffle이 없어서 같은 seed로 시작하면 항상 같은 순서로 읽혀요.

Chapter 4 Model

Embedding: 숫자에서 의미로

Token ID는 단순한 정수에요. 42번 token이 43번보다 "크다"는 수학적 의미는 없어요. Embedding은 이 이산적인 ID를 연속적인 고차원 벡터로 변환해서 신경망이 의미를 학습할 수 있는 공간으로 옮겨주는 과정이에요.

▶ 1

Embedding이란: Lookup Table

행렬의 행을 꺼내는 것

Embedding은 본질적으로 큰 행렬에서 해당 행을 꺼내는 것이에요. Token ID가 42이면, 1024x512 행렬의 42번째 행이 그 token의 벡터 표현이에요.

self.tok_emb = nn.Embedding(1024, 512) x = self.tok_emb(input_ids) # (batch, seq_len) → (batch, seq_len, 512) x = F.rms_norm(x, (x.size(-1),)) # normalize
▶ 2

왜 512차원인가: 표현력과 크기의 균형

model_dim 선택

model_dim=512는 각 token을 512개의 숫자로 표현한다는 뜻이에요. dim을 1024로 두 배 늘리면 parameter가 약 4배로 늘어서 16MB를 초과해요.

직관: 512차원은 "각 단어를 512가지 특성으로 설명하는 것"이에요. "감정 극성", "격식 수준", "시제 관련성" 같은 추상적인 축이 512개 있다고 생각할 수 있어요. 실제 축의 의미는 모델이 스스로 학습해요.
▶ 3

Embedding 이후: Residual Stream의 시작

transformer의 정보 고속도로

편지에 여러 사람이 메모를 붙이는 상황과 비슷해요. 원본 편지를 수정하지 않고, 각 사람이 자기의 해석이나 주석을 포스트잇으로 덧붙여요. 9개 block이 순서대로 자신의 "해석"을 추가하고, 마지막에 이 모든 정보가 합쳐진 상태에서 다음 token을 예측해요.

x = self.tok_emb(input_ids) x = F.rms_norm(x, (x.size(-1),)) x0 = x # 원본 보존 (resid_mix에서 참조) for block in self.blocks: x = block(x, x0) # 내부에서 residual add x = self.final_norm(x) logits = F.linear(x, self.tok_emb.weight) # tied head
Chapter 5 Core Mechanism

Attention: 문맥을 이해하는 메커니즘

같은 단어도 문맥에 따라 의미가 달라져요. Attention은 각 token이 주변 token들을 참조해서 자신의 표현을 문맥에 맞게 조정하는 메커니즘이에요.

▶ 1

Query, Key, Value: 정보 검색의 비유

attention의 세 가지 요소

도서관 검색에 비유할 수 있어요. Query는 "나는 이런 정보를 찾고 있어요"라는 질문이에요. Key는 각 책의 색인 카드로, "나는 이런 내용을 갖고 있어요"라는 정보에요. Value는 실제 책의 내용이에요. Query와 Key가 잘 맞는 책을 찾으면, 그 책의 Value를 가져오는 것이에요.

q = self.c_q(x) # Query: "나는 무엇을 찾는가" k = self.c_k(x) # Key: "나는 무엇을 가졌는가" v = self.c_v(x) # Value: "나의 실제 정보" # Q와 K의 dot product → attention score # score가 높은 위치의 V를 더 많이 참조
▶ 2

Multi-Head & Group Query Attention

여러 관점, 효율적으로

512차원 벡터를 8개의 head로 나눠요. 각 head는 64차원 공간에서 독립적으로 attention을 수행해요. GQA를 적용해서 Query head 8개가 KV head 4개를 공유해요.

KV projection의 output dim이 512에서 256으로 줄어서 parameter 50%를 절감하면서도 성능 저하는 거의 없어요.

▶ 3

RoPE: 위치 정보 주입

어디에 있는지 알려주기

Attention 자체는 token의 순서를 모르는에요. RoPE: Rotary Position Embedding은 Q와 K 벡터를 position에 따라 다른 각도로 회전시켜서 상대적 거리를 attention score에 반영해요.

def apply_rotary_emb(x, cos, sin): half = x.size(-1) // 2 x1, x2 = x[..., :half], x[..., half:] return torch.cat( (x1 * cos + x2 * sin, x1 * (-sin) + x2 * cos), dim=-1)
▶ 4

Causal Mask: 미래를 보지 못하게 하기

autoregressive의 핵심 제약

Language model은 왼쪽에서 오른쪽으로 생성해요. Causal mask는 각 token이 자신과 이전 token만 참조할 수 있도록 제한해요.

Check Your Understanding

GQA에서 Query head 8개, KV head 4개일 때, KV projection의 parameter 절감 비율은?

Standard MHA에서 K, V projection은 각각 dim=512 output이에요. GQA에서 KV head를 4개로 줄이면 output이 4*64=256이 돼요. 512에서 256으로, 즉 50% 절감이에요.
Chapter 6 Architecture

전체 Transformer Block: 모든 것의 조립

지금까지 배운 요소들이 하나의 Block 안에서 어떻게 조립되는지 살펴봐요. 이 프로젝트의 Block은 standard transformer와 약간 다른 설계를 채택하고 있어서, 각 차이점의 의도를 함께 이해해요.

▶ 1

Block 구조 전체 흐름

Norm → Attn → Norm → MLP
class Block(nn.Module): def forward(self, x, x0): # 1. Residual mixing mix = self.resid_mix x = mix[0] * x + mix[1] * x0 # 2. Attention sub-block n = self.attn_norm(x) attn_out = self.attn(n) x = x + self.attn_scale * attn_out # 3. MLP sub-block mlp_out = self.mlp(self.mlp_norm(x)) x = x + self.mlp_scale * mlp_out return x

최종 출력: Logits과 Cross-Entropy Loss

예측에서 학습 신호까지
x = self.final_norm(x) logits = F.linear(x, self.tok_emb.weight) logits = 30.0 * torch.tanh(logits / 30.0) # softcap loss = F.cross_entropy( logits.float().reshape(-1, 1024), target_ids.reshape(-1), reduction="mean")

Cross-entropy loss는 "모델의 예측 확률 분포가 정답과 얼마나 다른지"를 측정해요. 이 loss를 줄이는 방향으로 모든 parameter가 조정되는 것이 학습의 전체 과정이에요.

Final Check

이 모델의 전체 데이터 흐름을 올바르게 나열한 것은?

Token IDs가 embedding을 거쳐 vector가 되고, 9개의 transformer block을 통과하면서 각 block에서 RMSNorm, Attention, RMSNorm, MLP를 거쳐요. 마지막에 final RMSNorm과 tied LM head를 통해 logits가 나오고, target과의 cross-entropy loss가 계산돼요.
여기까지 요약: 텍스트가 BPE tokenizer로 정수 ID가 되고, binary shard에 저장돼요. 학습 시 TokenStream이 순차적으로 읽고, 각 GPU에 분배돼요. Token ID는 embedding으로 512-dim vector가 되고, 9개 transformer block을 통과하면서 attention으로 문맥을 이해하고 MLP로 정보를 가공해요. 최종적으로 다음 token의 확률을 예측하고, cross-entropy loss로 학습 신호를 만들어요. 이 과정이 20,000번 반복되면서 모델이 점점 좋은 예측을 하게 돼요.
Chapter 7 Optimization

Multi-Optimizer 전략: Muon과 Adam

일반적인 학습에서는 하나의 optimizer가 모든 parameter를 업데이트해요. 이 프로젝트는 parameter의 종류에 따라 서로 다른 optimizer를 사용하는 전략을 취해요. 2D matrix에는 Muon, embedding과 scalar에는 Adam을 쓰는 이유와 원리를 살펴볼게요.

▶ 1

왜 optimizer를 나누나요?

parameter 특성 차이

모델의 parameter는 크게 세 종류에요. 2D matrix는 attention의 Q/K/V projection이나 MLP의 linear layer처럼 (input_dim, output_dim) 형태의 큰 행렬이에요. Embedding은 token ID를 vector로 변환하는 lookup table이에요. Scalar/vector는 attn_scale, mlp_scale, resid_mix 같은 저차원 제어 parameter에요.

각각의 특성이 다르기 때문에, 한 가지 optimizer로 모두 최적화하면 learning rate나 momentum이 어느 한쪽에 맞지 않아요. 이 프로젝트에서는 각 유형에 맞는 optimizer와 learning rate를 따로 설정해요.

Parameter 유형OptimizerLearning Rate
2D matrixMuon0.04
Token embeddingAdam0.05 (tied)
LM head (untied 시)Adam0.008
Scalar / vectorAdam0.04
▶ 2

Adam: 적응형 학습률의 표준

1차 모멘트 + 2차 모멘트

Adam은 gradient의 1차 모멘트인 평균과 2차 모멘트인 분산을 추적하면서, 각 parameter마다 적응적으로 learning rate를 조절하는 optimizer에요. Gradient가 일관되게 같은 방향이면 크게 움직이고, 불안정하면 작게 움직여요.

이 프로젝트에서 Adam은 embedding, scalar, vector parameter에 사용돼요. 이런 parameter들은 차원이 낮거나 특수한 역할을 하므로, Adam의 적응적 특성이 잘 맞아요.

optimizer_tok = torch.optim.Adam( [{"params": [base_model.tok_emb.weight], "lr": token_lr, "base_lr": token_lr}], betas=(0.9, 0.95), eps=1e-8, fused=True) optimizer_scalar = torch.optim.Adam( [{"params": scalar_params, "lr": 0.04, "base_lr": 0.04}], betas=(0.9, 0.95), eps=1e-8, fused=True)
▶ 3

Muon: 행렬 전용 optimizer

Newton-Schulz 직교화

Muon은 2D matrix parameter 전용으로 설계된 optimizer에요. 핵심 아이디어는 gradient를 직교 행렬로 변환한 뒤 적용하는 것이에요. 직교 행렬은 모든 singular value가 1인 행렬로, 업데이트의 방향만 유지하고 크기를 균일하게 만들어요.

일반적인 optimizer가 "이 방향으로 이만큼 가라"고 지시한다면, Muon은 "이 방향으로 가되, 모든 차원에서 동일한 보폭으로 가라"고 지시하는 것에 가까워요. 이렇게 하면 특정 방향으로 지나치게 큰 업데이트가 일어나는 것을 방지해요.

def zeropower_via_newtonschulz5(G, steps=10, eps=1e-7): # Newton-Schulz 반복으로 행렬을 직교화 a, b, c = (3.4445, -4.7750, 2.0315) X = G.bfloat16() X /= X.norm() + eps for _ in range(steps): A = X @ X.T B = b * A + c * A @ A X = a * X + B @ X return X

Newton-Schulz iteration은 SVD를 계산하지 않고도 행렬을 직교화하는 빠른 방법이에요. 5번의 반복으로 충분히 좋은 근사치를 얻을 수 있어서, SVD 대비 훨씬 효율적이에요.

▶ 4

Muon의 분산 처리와 momentum

GPU 간 작업 분배

Newton-Schulz 직교화는 연산 비용이 있어요. Muon은 이를 여러 GPU에 분산시켜요. 각 GPU가 전체 parameter 중 자기 몫만 직교화한 뒤, all_reduce로 결과를 합쳐요.

# 각 GPU가 자기 몫의 parameter만 처리 for i, p in enumerate(params): if i % world_size == rank and p.grad is not None: buf.mul_(momentum).add_(g) # momentum 적용 g = g.add(buf, alpha=momentum) # Nesterov g = zeropower_via_newtonschulz5(g) # 직교화 g *= max(1, g.size(0) / g.size(1)) ** 0.5 # 결과를 모든 GPU에 합산 dist.all_reduce(updates_flat, op=dist.ReduceOp.SUM)

Momentum도 0.85에서 시작해 500 step에 걸쳐 0.95까지 서서히 올려요. 초반에 momentum이 너무 높으면 학습이 불안정해질 수 있어서 이렇게 warmup을 적용해요.

Check Your Understanding

Muon optimizer에서 Newton-Schulz 직교화가 하는 역할은?

Newton-Schulz iteration은 행렬의 singular value를 모두 1로 만드는 "zeropower" 연산이에요. 이렇게 하면 gradient의 방향 정보만 남기고 크기를 균일화해서, 특정 방향으로 과도한 업데이트가 일어나는 것을 방지해요.
Chapter 8 Training

Training Loop: 20,000번의 반복

모델 구조와 optimizer가 준비되었으니, 이제 실제로 학습이 어떻게 진행되는지 살펴볼게요. Gradient accumulation, learning rate scheduling, 분산 학습, wallclock 제한 등 이 프로젝트 training loop의 핵심 요소를 하나씩 풀어볼게요.

▶ 1

Gradient Accumulation: 작은 GPU로 큰 batch

micro-step 합산

이 프로젝트는 한 step당 524,288개의 token을 처리해요. 하지만 하나의 GPU에 이 전체를 한 번에 올릴 수 없으므로, 여러 micro-step으로 나눠서 처리해요. 각 micro-step에서 작은 batch의 gradient를 계산하고 누적한 뒤, 모든 micro-step이 끝나면 한 번에 parameter를 업데이트해요.

grad_accum_steps = 8 // world_size # GPU 수에 따라 조정 grad_scale = 1.0 / grad_accum_steps for micro_step in range(grad_accum_steps): x, y = train_loader.next_batch(...) with torch.autocast(dtype=torch.bfloat16): loss = model(x, y) (loss * grad_scale).backward() # gradient 누적 for opt in optimizers: opt.step() # 한 번에 업데이트

시험 공부에 비유하면, 한 챕터 전체를 한 번에 읽고 정리하는 대신, 섹션별로 읽으면서 메모를 모아두고, 챕터가 끝나면 모인 메모를 종합해서 노트를 갱신하는 것과 같아요.

▶ 2

Warmup과 Warmdown: 학습률 스케줄

시작과 끝을 부드럽게

학습 초반에는 parameter가 랜덤 상태이므로, 큰 learning rate를 바로 적용하면 불안정해질 수 있어요. 이 프로젝트는 warmup 20 step 동안 컴파일된 forward/backward 경로를 준비하고, 학습 마지막 1,200 step에서는 warmdown으로 learning rate를 점진적으로 0으로 줄여요.

def lr_mul(step, elapsed_ms): # warmdown: 남은 시간 기반으로 lr을 점진적으로 줄임 remaining_ms = max_wallclock_ms - elapsed_ms warmdown_ms = warmdown_iters * step_ms if remaining_ms <= warmdown_ms: return remaining_ms / warmdown_ms return 1.0
시간 기반 warmdown: 이 프로젝트의 warmdown은 step 수가 아니라 남은 wallclock 시간을 기준으로 동작해요. 10분 제한 안에서 최대한 학습하되, 끝이 가까워지면 자연스럽게 수렴하도록 설계된 것이에요.
▶ 3

분산 학습과 DDP

여러 GPU 동기화

여러 GPU를 사용할 때, PyTorch의 DistributedDataParallel: DDP이 gradient 동기화를 담당해요. 각 GPU가 서로 다른 데이터로 forward/backward를 수행하고, backward의 마지막 micro-step에서만 gradient를 동기화해요.

# 마지막 micro-step에서만 gradient 동기화 model.require_backward_grad_sync = ( micro_step == grad_accum_steps - 1)

이렇게 하면 중간 micro-step의 불필요한 통신을 줄여서 학습 속도를 높여요. Wallclock 제한이 10분인 이 프로젝트에서 통신 비용 절감은 매우 중요해요.

▶ 4

Wallclock 제한과 Early Stopping

10분 안에 끝내기

이 프로젝트는 최대 20,000 iteration이지만, 실제로는 600초인 10분의 wallclock 제한이 먼저 도달해요. 시간 제한에 도달하면 학습을 중단하고 마지막 validation을 수행해요.

reached_cap = approx_training_time_ms >= max_wallclock_ms if distributed: # 모든 GPU가 동의해야 멈춤 dist.all_reduce(reached_cap_tensor, op=dist.ReduceOp.MAX) if reached_cap: stop_after_step = step # 다음 step에서 정지

분산 환경에서는 모든 GPU 중 하나라도 시간 제한에 도달하면 전체가 멈춰요. all_reduce로 MAX를 취해서 가장 빠르게 제한에 도달한 GPU의 판단을 따르는 것이에요.

Check Your Understanding

Gradient accumulation에서 각 micro-step의 loss에 grad_scale을 곱하는 이유는?

각 micro-step에서 loss * (1/grad_accum_steps)를 backward하면, 누적된 gradient는 전체 batch에 대한 평균 gradient와 동일해져요. 이렇게 해야 gradient accumulation 유무와 관계없이 같은 학습 동작을 보장할 수 있어요.
Chapter 9 Compression

int8 Quantization과 zlib 압축

학습이 끝나면 모델을 제출해야 해요. 이 프로젝트의 제출 크기 제한은 16MB에요. fp32 모델은 이 제한을 훨씬 초과하므로, int8 양자화와 zlib 압축을 거쳐야 해요. 핵심은 정밀도 손실을 최소화하면서 크기를 극적으로 줄이는 것이에요.

▶ 1

int8 양자화란?

32-bit → 8-bit

양자화는 높은 정밀도의 숫자를 낮은 정밀도로 변환하는 과정이에요. fp32 weight는 한 값당 4 byte를 차지하지만, int8로 변환하면 1 byte로 줄어요. 이론적으로 4배 압축이에요.

수채화 그림을 떠올려 보세요. 원본은 수백만 가지 색을 쓰지만, 제한된 팔레트인 256색으로 다시 그려도 눈으로 보기에 큰 차이가 없는 것과 비슷해요. 각 색의 미세한 차이는 사라지지만, 전체적인 그림은 유지돼요.

▶ 2

Per-Row Scale: 행별 정밀도 보존

2D matrix 전략

단순히 전체 행렬에 하나의 scale을 적용하면, 값의 범위가 행마다 다를 때 정밀도가 크게 떨어져요. 이 프로젝트는 2D matrix에 per-row scale을 사용해요. 각 행마다 독립적인 scale factor를 두어서, 행별 값 범위 차이를 보존해요.

def quantize_float_tensor(t32): if t32.ndim == 2: # 행별로 quantile 클리핑 후 scale 계산 clip_abs = torch.quantile( t32.abs(), INT8_CLIP_Q, dim=1) scale = (clip_abs / 127.0).clamp_min(1.0/127.0) q = torch.round(clipped / scale[:, None]) return q.to(torch.int8), scale

INT8_CLIP_Q는 quantile 기반 clipping으로, 극단적인 outlier를 제거해서 나머지 값들의 정밀도를 높여요. 이 프로젝트에서는 0.9999 quantile을 사용해요.

▶ 3

작은 tensor는 양자화하지 않는다

fp16 passthrough

모든 tensor를 int8로 바꾸는 건 아니에요. 원소 수가 적은 tensor는 양자화 오차가 상대적으로 크고, int8 + scale의 오버헤드가 오히려 손해에요. 이런 tensor는 fp16으로 캐스팅해서 그대로 저장해요.

판단 기준: INT8_KEEP_FLOAT_MAX_NUMEL 이하의 원소를 가진 tensor는 양자화 없이 fp16으로 저장돼요. attn_scale, mlp_scale 같은 scalar parameter가 여기에 해당해요.
▶ 4

zlib 압축과 Roundtrip 검증

최종 크기 줄이기

int8 양자화 후에도 추가로 zlib 압축을 적용해요. zlib는 데이터의 반복 패턴을 찾아서 무손실 압축하는 알고리즘이에요. 최종 파일 확장자 .int8.ptz는 이 양자화+압축 포맷을 나타내요.

quant_obj, quant_stats = quantize_state_dict_int8( base_model.state_dict()) quant_buf = io.BytesIO() torch.save(quant_obj, quant_buf) quant_blob = zlib.compress(quant_buf.getvalue(), level=9) # Roundtrip 검증: 압축 해제 → 복원 → 다시 평가 quant_state = torch.load( io.BytesIO(zlib.decompress(quant_blob_disk))) base_model.load_state_dict( dequantize_state_dict_int8(quant_state))
Roundtrip 검증이 필수인 이유: 양자화는 손실 과정이에요. 압축된 모델을 다시 복원했을 때 성능이 얼마나 떨어지는지 반드시 확인해야 해요. 이 프로젝트는 roundtrip 후 val_loss와 val_bpb를 재측정해요.

Check Your Understanding

이 프로젝트에서 per-row scale을 사용하는 이유는?

2D matrix의 각 행은 서로 다른 output channel에 대응하며, 값의 분포가 행마다 다를 수 있어요. per-tensor scale은 전체에서 가장 큰 값에 맞추므로 작은 값들의 정밀도가 떨어지지만, per-row scale은 각 행의 범위에 맞춰서 모든 행의 정밀도를 균일하게 유지해요.
Chapter 10 Evaluation

bits-per-byte: 공정한 평가 지표

모델이 얼마나 "잘" 학습되었는지 어떻게 측정할까요? Cross-entropy loss만으로는 서로 다른 tokenizer를 쓰는 모델을 비교할 수 없어요. 이 프로젝트는 bits-per-byte라는 tokenizer에 독립적인 지표를 사용해요.

▶ 1

왜 loss만으로는 부족한가

tokenizer 의존성 문제

Cross-entropy loss는 "token당 평균 정보 손실"을 측정해요. 그런데 같은 텍스트라도 vocab=1024인 tokenizer는 vocab=100K보다 훨씬 많은 token을 만들어요. Token이 많으면 개별 예측이 쉬워지므로 loss가 낮아 보이지만, 이건 모델이 뛰어난 게 아니라 tokenizer가 세밀한 것일 뿐이에요.

공정한 비교의 핵심: 모델 성능을 tokenizer 설계와 분리해야 해요. bits-per-byte는 원본 텍스트의 byte 단위로 정보 효율을 측정하므로, tokenizer가 달라도 공정하게 비교할 수 있어요.
▶ 2

bits-per-byte 계산 원리

nats → bits → bytes

bits-per-byte는 세 단계로 계산돼요. 먼저 cross-entropy loss를 nat 단위에서 bit 단위로 변환하고, token당 bit를 byte당 bit로 변환해요.

# 1. token별 loss 합산 loss_sum += per_token_loss.sum() # 2. token이 나타내는 byte 수 합산 tok_bytes = base_bytes_lut[target_ids] tok_bytes += has_leading_space_lut[target_ids] \ & ~is_boundary_token_lut[prev_ids] byte_sum += tok_bytes.sum() # 3. bits-per-byte = total_loss_in_bits / total_bytes val_bpb = (loss_sum / ln(2)) / byte_sum

base_bytes_lut는 Ch.2에서 만든 lookup table이에요. 각 token이 원본 텍스트에서 몇 byte를 차지하는지 기록해요. 공백 처리도 섬세해요. SentencePiece의 ▁ 기호가 나타내는 선행 공백 1 byte를 정확히 반영해요.

▶ 3

bits-per-byte의 직관적 의미

압축 효율의 척도

bits-per-byte가 1.0이면, 모델이 원본 텍스트를 8분의 1 크기로 압축할 수 있다는 뜻이에요. 원래 byte당 8 bit가 필요한데 1 bit만 있으면 된다는 것이니까요. 값이 낮을수록 모델이 텍스트를 더 잘 "이해"하고 있다는 의미에요.

bits-per-byte의미
8.0예측 없음. 완전한 무작위 추측
~5.0기초적인 문자 빈도 수준의 예측
~1.0매우 좋은 language model
< 0.8최신 대형 모델 수준

Check Your Understanding

Vocab=1024인 모델 A와 vocab=32000인 모델 B가 같은 cross-entropy loss를 기록했어요. bits-per-byte는 어떤 모델이 더 낮을까요?

같은 loss라도, vocab=32000인 모델 B의 token은 더 많은 byte를 나타내요. bits-per-byte는 (loss_in_bits * tokens) / total_bytes이므로, token당 더 많은 byte를 커버하는 B의 bits-per-byte가 더 낮아요. 이것이 바로 loss만으로 비교하면 안 되는 이유이기도 해요.
Chapter 11 Advanced

Test-Time Training: per-document LoRA 적응

보통 모델은 학습이 끝나면 weight가 고정되고, 평가 시에는 그대로 사용해요. 이 프로젝트는 한 발 더 나아가서, 평가 시에도 각 문서에 맞춰 모델을 미세 조정해요. 이 기법을 Test-Time Training: TTT이라 부르고, LoRA를 사용해서 효율적으로 구현해요.

▶ 1

LoRA: 적은 parameter로 적응하기

Low-Rank Adaptation

LoRA: Low-Rank Adaptation은 기존 weight를 고정하고, 작은 추가 행렬만 학습해서 모델을 적응시키는 기법이에요. 원래 weight matrix W에 저차원 행렬 B와 A의 곱인 ΔW = BA를 더해요. A의 크기는 (rank, input_dim), B는 (output_dim, rank)로 rank가 작으면 추가 parameter가 매우 적어요.

▶ 2

문서별 독립 적응: Batched LoRA

각 문서마다 별도 adapter

이 프로젝트에서 TTT의 핵심은 문서 경계를 기준으로 각 문서마다 독립적인 LoRA adapter를 학습하는 것이에요. BOS token으로 문서 경계를 식별하고, 각 문서를 chunk로 나눠서 순차적으로 적응해요.

class BatchedTTTLoRA(nn.Module): # 각 batch element마다 독립적인 LoRA adapter def __init__(self, bsz, model, rank): self.lm_head_lora = BatchedLinearLoRA( bsz, dim, vocab, rank) self.q_loras = nn.ModuleList() self.v_loras = nn.ModuleList() for block in model.blocks: self.q_loras.append( BatchedLinearLoRA(...)) # Q projection self.v_loras.append( BatchedLinearLoRA(...)) # V projection

Adapter는 LM head, 그리고 각 block의 Q와 V projection에 적용돼요. K에는 적용하지 않아요. Q와 V만으로도 충분한 적응 능력을 확보하면서 parameter를 절약해요.

▶ 3

Chunk 단위 학습과 평가 흐름

score-then-train

각 문서를 256 token 크기의 chunk로 나눠요. 매 chunk에서의 순서는 "먼저 점수를 매기고, 그 다음 학습"이에요. 현재 chunk에 대한 loss를 bits-per-byte 계산에 기록한 뒤, 그 chunk의 loss로 LoRA adapter를 업데이트해요. 이렇게 하면 다음 chunk 예측 시 이전 chunk에서 배운 패턴을 활용할 수 있어요.

Score-then-train: 한 chunk의 loss를 평가에 사용한 뒤 학습에도 사용해요. 평가와 학습이 분리된 것이 아니라, 평가 직후 같은 데이터로 학습해서 다음 chunk 예측을 개선해요. 마지막 chunk는 학습 없이 평가만 해요. 이미 이후에 활용할 chunk가 없으니까요.
▶ 4

TTT가 중요한 이유: 대회 점수 산출

최종 val_bpb

이 프로젝트에서 TTT LoRA의 결과가 최종 대회 점수에요. 학습 완료 후 int8+zlib roundtrip을 거친 모델에 TTT LoRA를 적용해서 측정한 val_bpb가 최종 제출 점수에요.

# 최종 대회 점수 산출 torch._dynamo.reset() ttt_val_loss, ttt_val_bpb = eval_val_ttt_lora( args, base_model, rank, world_size, device, base_bytes_lut, has_leading_space_lut, is_boundary_token_lut)

기본 모델의 val_bpb보다 TTT를 적용한 val_bpb가 더 낮아요. 문서별 적응이 예측 성능을 실질적으로 개선한다는 증거에요.

Final Check

TTT LoRA에서 마지막 chunk는 학습 없이 평가만 하는 이유는?

TTT의 학습은 "다음 chunk를 더 잘 예측하기 위한" 것이에요. 마지막 chunk 이후에는 예측할 데이터가 없으므로, 학습하는 것이 무의미해요. 이때 gradient graph를 유지하지 않으므로 메모리도 절약돼요.
전체 파이프라인 요약: 텍스트가 BPE tokenizer로 정수 ID가 되고, binary shard에 저장돼요. 학습 시 TokenStream이 데이터를 읽고 GPU에 분배해요. Token ID는 embedding으로 vector가 되고, 9개 transformer block을 통과해요. Muon과 Adam이 각각 matrix와 scalar parameter를 최적화해요. 20,000 step 또는 10분 후 학습이 끝나면, int8 양자화 + zlib 압축으로 16MB 이하로 줄여요. 최종 평가는 양자화된 모델에 per-document LoRA TTT를 적용해서 bits-per-byte를 측정해요.
