Step-by-Step Learning Guide — From Tokenizer to Transformer Training

Chapter 1 Foundation

Tokenizer: The First Gate to Converting Text into Numbers

Computers don't understand text directly. Before a neural network can process text, we first need to convert it into a sequence of numbers. That's what a tokenizer does. The tokenizer's design determines how many parameters go to embeddings versus actual learning, and how long each input sequence gets.

▶ 1

What is a Tokenizer?

text → numbers

A tokenizer breaks text strings into units called tokens, and assigns each token a unique integer ID. For example, the string "Hello world" becomes an integer array like [15496, 995] after tokenization.

The reverse process is also possible. Converting an integer array back to text is called decoding. A token can be a whole word, part of a word, a single character, or even a byte. Deciding what unit to split on is the core design decision in tokenizer architecture.

▶ 2

Why Not Split by Words?

the vocabulary size problem

The most intuitive approach is to split words based on whitespace. But this approach has a serious problem.

Even just in English, there are hundreds of thousands of words, and when you include proper nouns, technical terms, and neologisms, the list becomes practically infinite. If your vocabulary has a million words, your model needs an embedding table of size 1,000,000 x dim. With a 16MB constraint like we have in this project, the embedding table consumes most of the budget, leaving almost no parameters for actual learning.

If we split into individual characters or bytes instead, the vocabulary shrinks but sequence length explodes. If "Hello" becomes 5 tokens, the model has to process much longer sequences. Since attention computation scales with the square of sequence length, costs skyrocket.

Core trade-off: A large vocabulary means a bigger embedding table, but a small vocabulary means longer sequences. The whole game in tokenizer design is finding the right balance between these two.
▶ 3

BPE: Byte Pair Encoding

the algorithm behind modern tokenizers

Byte Pair Encoding is the most widely-used algorithm for solving this trade-off. GPT, LLaMA, and this project all use BPE-based tokenizers.

BPE is simple in principle. First, we decompose the text into the smallest units: individual bytes or characters. Then, we repeatedly identify the most frequent consecutive pair in the training data and merge them into a new token.

Think of it like letter blocks. At first, you only have blocks with single letters. When "t" and "h" frequently appear side by side, you create a new "th" block by combining them. When "th" and "e" frequently stick together, you create a "the" block too. You repeat this process until you reach your target vocabulary size.

# Initial state: decompose all text into individual bytes corpus = "the cat sat on the mat" tokens = ['t', 'h', 'e', ' ', 'c', 'a', 't', ...] # Step 1: Find the most frequent pair → ('t', 'h') is most common # → Add new token 'th', vocab = [..., 'th'] # Step 2: Now ('th', 'e') is most frequent # → Add new token 'the', vocab = [..., 'th', 'the'] # Step 3: (' ', 'the') is frequent # → Add new token ' the' # Repeat until reaching desired vocab size

So common words and word fragments become single tokens, while rare words get broken into multiple tokens. This keeps vocabulary size within our 1024-token budget while still capturing frequent patterns efficiently.

▶ 4

SentencePiece: A Practical Implementation of BPE

this project's tokenizer library

SentencePiece is a tokenizer library created by Google that actually implements the BPE algorithm. In this project, we use it with import sentencepiece as spm.

The key feature of SentencePiece is that it doesn't pre-split text by whitespace. Instead, it treats spaces like any other character, but adds a special symbol ▁ at word boundaries. For example, "Hello world" becomes ["▁Hello", "▁world"]. This ▁ symbol means "there was a space before this token".

Special tokens: Beyond normal text tokens, SentencePiece supports control tokens. There are special tokens like BOS (Beginning of Sequence), EOS (End of Sequence), and UNK (Unknown). In this project, we use the BOS token to mark document boundaries.

Check Your Understanding

What happens when you increase the vocabulary size of a BPE tokenizer?

When vocabulary grows, longer words and phrases can be represented as single tokens, so sequences get shorter. But the embedding table gains more rows, so the table itself gets bigger. We deliberately set vocab=1024, very small, because of the 16MB size constraint.
Chapter 2 Project-Specific

Project Tokenizer: vocab_size = 1024

We set vocab_size to 1024. Compared to GPT-4's 100,000+ and LLaMA's 32,000, that's about 1% of what most models use. Let's look at how this choice impacts the overall model design.

▶ 2

Tied Embedding: One Matrix for Two Jobs

input & output sharing

Usually, a language model has two large matrices. One is the input embedding, which converts token IDs to vectors. The other is the output head, which converts vectors back into a probability distribution over vocabulary. Tied embedding is a technique where these two matrices share the same weights.

This cuts the parameter count in half and keeps input and output token representations aligned.

▶ 3

Bits-Per-Byte: A Tokenizer-Independent Evaluation Metric

the key to fair comparison

How do we fairly compare models with different vocabulary sizes? We use bits-per-byte. It measures how well the model "compresses" text by counting the number of bits per byte in the original text needed to achieve that compression.

# Computing bits-per-byte loss_in_nats = 2.35 bits_per_token = loss_in_nats / ln(2) # = 3.39 tokens_per_byte = total_tokens / total_bytes bits_per_byte = bits_per_token * tokens_per_byte

▶ 2

TokenStream: Endless Data Flow

sequential read + auto-wrap

Imagine multiple books sitting on a shelf. TokenStream is like a reading robot that starts with the first book, reads to the end, automatically moves to the next book, and when it reaches the last book, it loops back to the first and keeps reading endlessly.

class TokenStream: def take(self, n): chunks = [] remaining = n while remaining > 0: avail = self.tokens.numel() - self.pos if avail <= 0: self._advance_file() # Move to next shard continue k = min(remaining, avail) chunks.append(self.tokens[self.pos : self.pos + k]) self.pos += k remaining -= k return torch.cat(chunks)

This design is deterministic. There's no random shuffling, so if you start with the same seed you always read in the same order.

Chapter 4 Model

Embedding: From Numbers to Meaning

A token ID is just a simple integer. Token 42 isn't mathematically "larger" than token 43. Embedding transforms these discrete IDs into continuous high-dimensional vectors, creating a space where the neural network can learn meaningful relationships.

▶ 1

What is Embedding: A Lookup Table

Extracting a row from a matrix

Embedding is fundamentally extracting a row from a large matrix. If a token ID is 42, the 42nd row of our 1024x512 matrix becomes that token's vector representation.

self.tok_emb = nn.Embedding(1024, 512) x = self.tok_emb(input_ids) # (batch, seq_len) → (batch, seq_len, 512) x = F.rms_norm(x, (x.size(-1),)) # normalize
▶ 2

Why 512 Dimensions: Balancing Expressiveness and Size

Choosing model_dim

model_dim=512 means we represent each token using 512 numbers. If we doubled the dimension to 1024, parameters would grow by about 4x, exceeding 16MB.

Intuition: 512 dimensions means describing each word across 512 different properties. Think of 512 abstract axes—things like "emotional tone," "formality level," "temporal relevance." The model learns what these axes actually mean.
▶ 3

After Embedding: The Start of the Residual Stream

The transformer's information highway

Think of it like a letter being passed around. Multiple people add sticky notes without modifying the original. Each person contributes their own interpretation or annotation. Our 9 blocks add their "interpretation" one by one, and at the end, all this accumulated information guides the prediction of the next token.

x = self.tok_emb(input_ids) x = F.rms_norm(x, (x.size(-1),)) x0 = x # preserve original (referenced in resid_mix) for block in self.blocks: x = block(x, x0) # internal residual addition x = self.final_norm(x) logits = F.linear(x, self.tok_emb.weight) # tied head
Chapter 5 Core Mechanism

Attention: The Mechanism for Understanding Context

The same word means different things in different contexts. Attention is the mechanism that lets each token look at its neighbors and adjust its representation to match the surrounding context.

▶ 1

Query, Key, Value: A Library Search Analogy

The three components of attention

Think of searching a library. A Query is the question: "I'm looking for this kind of information." A Key is like a catalog card for each book—"I contain this kind of content." Value is the actual book content. When the query matches a key well, we take that book's value.

q = self.c_q(x) # Query: "what am I looking for" k = self.c_k(x) # Key: "what do I have" v = self.c_v(x) # Value: "my actual information" # dot product of Q and K → attention score # higher scores mean we reference that position's V more
▶ 2

Multi-Head & Group Query Attention

Multiple perspectives, efficiently

We split the 512-dimensional vector across 8 heads. Each head independently performs attention in 64-dimensional space. With GQA applied, 8 query heads share 4 key-value heads.

The KV projection shrinks from 512 to 256 output dimensions, cutting parameters in half while barely hurting performance.

▶ 3

RoPE: Injecting Position Information

Telling the model where things are

Attention by itself doesn't know token order. RoPE (Rotary Position Embedding) rotates Q and K vectors at different angles based on position, encoding relative distance into attention scores.

def apply_rotary_emb(x, cos, sin): half = x.size(-1) // 2 x1, x2 = x[..., :half], x[..., half:] return torch.cat( (x1 * cos + x2 * sin, x1 * (-sin) + x2 * cos), dim=-1)
▶ 4

Causal Mask: Preventing Peeking at the Future

The core constraint of autoregressive generation

Language models generate left to right. A causal mask restricts each token to attend only to itself and prior tokens.

Check Your Understanding

In GQA with 8 query heads and 4 KV heads, what's the parameter reduction in KV projection?

In standard multi-head attention, K and V projections each produce dim=512 outputs. With GQA using 4 KV heads, the output becomes 4*64=256. That's a reduction from 512 to 256, which is 50%.

Chapter 7 Optimization

Multi-Optimizer Strategy: Muon and Adam

In typical training, a single optimizer updates all parameters. We use a different strategy here: different optimizers for different parameter types. We'll look at why and how we use Muon for 2D matrices and Adam for embeddings and scalars.

▶ 1

Why split optimizers?

Parameter type differences

Model parameters fall into three categories. 2D matrices are large tensors like (input_dim, output_dim) used in attention Q/K/V projections or MLP linear layers. Embeddings are lookup tables that map token IDs to vectors. Scalars and vectors are low-dimensional control parameters like attn_scale, mlp_scale, and resid_mix.

Since each type has different characteristics, using a single optimizer for all of them doesn't work well—the learning rate and momentum end up compromising one type or the other. We set separate optimizers and learning rates for each parameter type.

Parameter TypeOptimizerLearning Rate
2D matrixMuon0.04
Token embeddingAdam0.05 (tied)
LM head (untied)Adam0.008
Scalar / vectorAdam0.04
▶ 2

Adam: The Standard Adaptive Learning Rate

First moment + second moment

Adam tracks the first moment (mean) and second moment (variance) of the gradient, adjusting the learning rate adaptively for each parameter. If the gradient consistently points in the same direction, we take a large step. If it's noisy, we take a small one.

In this project, Adam handles embeddings, scalars, and vectors. These parameters are either low-dimensional or have specialized roles, so Adam's adaptive nature fits them well.

optimizer_tok = torch.optim.Adam( [{"params": [base_model.tok_emb.weight], "lr": token_lr, "base_lr": token_lr}], betas=(0.9, 0.95), eps=1e-8, fused=True) optimizer_scalar = torch.optim.Adam( [{"params": scalar_params, "lr": 0.04, "base_lr": 0.04}], betas=(0.9, 0.95), eps=1e-8, fused=True)
▶ 3

Muon: Optimizer for Matrices

Newton-Schulz orthogonalization

Muon is an optimizer designed exclusively for 2D matrix parameters. The core idea is to convert the gradient into an orthogonal matrix before applying it. An orthogonal matrix has all singular values equal to 1, so it preserves the direction of the update while making the magnitude uniform across all dimensions.

While a typical optimizer says "move this far in that direction," Muon says "move in that direction, but take the same size step in every dimension." This prevents disproportionately large updates along certain directions.

def zeropower_via_newtonschulz5(G, steps=10, eps=1e-7): # Orthogonalize the matrix via Newton-Schulz iteration a, b, c = (3.4445, -4.7750, 2.0315) X = G.bfloat16() X /= X.norm() + eps for _ in range(steps): A = X @ X.T B = b * A + c * A @ A X = a * X + B @ X return X

Newton-Schulz iteration is a fast way to orthogonalize a matrix without computing the full SVD. Five iterations give us a good approximation, making it much more efficient than SVD.

▶ 4

Muon's Distributed Computation and Momentum

Work distribution across GPUs

Newton-Schulz orthogonalization has computational cost. Muon spreads this across multiple GPUs. Each GPU orthogonalizes its own slice of parameters, then we combine the results with all_reduce.

# Each GPU processes only its assigned parameters for i, p in enumerate(params): if i % world_size == rank and p.grad is not None: buf.mul_(momentum).add_(g) # apply momentum g = g.add(buf, alpha=momentum) # Nesterov g = zeropower_via_newtonschulz5(g) # orthogonalize g *= max(1, g.size(0) / g.size(1)) ** 0.5 # Sum results across all GPUs dist.all_reduce(updates_flat, op=dist.ReduceOp.SUM)

Momentum also starts at 0.85 and gradually increases to 0.95 over 500 steps. Too much momentum early on destabilizes training, so we warm it up.

Check Your Understanding

What is the role of Newton-Schulz orthogonalization in the Muon optimizer?

Newton-Schulz iteration performs "zeropower" to make all singular values equal to 1. This preserves the direction of the gradient while normalizing the magnitude, preventing excessive updates in any single direction.
Chapter 8 Training

Training Loop: 20,000 Iterations

With the model architecture and optimizer ready, let's see how training actually happens. We'll walk through gradient accumulation, learning rate scheduling, distributed training, and wallclock limits—the core elements of this project's training loop.

▶ 1

Gradient Accumulation: Large Batches on Small GPUs

Summing micro-steps

We process 524,288 tokens per step. But we can't fit all of that on a single GPU at once, so we break it into several micro-steps. Each micro-step computes gradients on a small batch and accumulates them. Once all micro-steps finish, we update parameters once.

grad_accum_steps = 8 // world_size # Adjust based on number of GPUs grad_scale = 1.0 / grad_accum_steps for micro_step in range(grad_accum_steps): x, y = train_loader.next_batch(...) with torch.autocast(dtype=torch.bfloat16): loss = model(x, y) (loss * grad_scale).backward() # accumulate gradients for opt in optimizers: opt.step() # update once

It's like studying a textbook chapter—instead of reading the whole thing at once and then taking notes, you read section by section, collecting notes as you go, then consolidate all your notes once the chapter is done.

▶ 2

Warmup and Warmdown: Learning Rate Scheduling

Smooth start and finish

Early in training, parameters are random, so applying a large learning rate immediately can be unstable. We do warmup for 20 steps to prepare the compiled forward/backward paths, and in the final 1,200 steps we warmdown, gradually reducing the learning rate to zero.

def lr_mul(step, elapsed_ms): # Warmdown: gradually reduce lr based on remaining time remaining_ms = max_wallclock_ms - elapsed_ms warmdown_ms = warmdown_iters * step_ms if remaining_ms <= warmdown_ms: return remaining_ms / warmdown_ms return 1.0
Time-based warmdown: Our warmdown isn't based on step count—it's based on remaining wallclock time. We train as much as we can within the 10-minute limit, but naturally converge as we approach the deadline.
▶ 3

Distributed Training and DDP

Syncing across multiple GPUs

When using multiple GPUs, PyTorch's DistributedDataParallel (DDP) handles gradient synchronization. Each GPU does forward/backward on its own data, and we only synchronize gradients on the last micro-step of backward.

# Sync gradients only on the last micro-step model.require_backward_grad_sync = ( micro_step == grad_accum_steps - 1)

This cuts down unnecessary communication between micro-steps, speeding up training. With only a 10-minute wallclock limit, reducing communication overhead is critical.

▶ 4

Wallclock Limits and Early Stopping

Finishing in 10 minutes

The config allows up to 20,000 iterations, but in practice we hit a 600-second (10-minute) wallclock limit first. Once we reach the time limit, we stop training and run final validation.

reached_cap = approx_training_time_ms >= max_wallclock_ms if distributed: # All GPUs must agree to stop dist.all_reduce(reached_cap_tensor, op=dist.ReduceOp.MAX) if reached_cap: stop_after_step = step # Stop after this step

In distributed settings, if any GPU reaches the time limit, all of them stop. We use all_reduce with MAX to have all GPUs follow the fastest one's decision.

Check Your Understanding

Why do we multiply each micro-step's loss by grad_scale in gradient accumulation?

When we backward loss * (1/grad_accum_steps) at each micro-step, the accumulated gradient becomes equivalent to the average gradient of the full batch. This ensures consistent learning behavior whether we use accumulation or not.
Chapter 9 Compression

int8 Quantization and zlib Compression

Once training finishes, we need to submit the model. The competition has a submission size limit of 16MB. An fp32 model far exceeds this, so we need int8 quantization and zlib compression. The goal is to shrink the model dramatically while losing as little precision as possible.

▶ 1

What is int8 Quantization?

32-bit to 8-bit

Quantization converts high-precision numbers to low-precision ones. An fp32 weight takes 4 bytes, but int8 takes 1 byte. That's a 4x compression in theory.

Think of a watercolor painting. The original uses millions of colors, but if you redraw it with a limited palette of 256 colors, it looks nearly the same to the eye. You lose subtle color gradations, but the overall picture stays intact.

▶ 2

Per-Row Scale: Preserving Precision Per Row

Strategy for 2D matrices

If we use a single scale factor for the entire matrix, we lose precision when value ranges differ across rows. We use per-row scale: each row gets its own scale factor, preserving differences in value ranges across rows.

def quantize_float_tensor(t32): if t32.ndim == 2: # Clip by quantile per row, then calculate scale clip_abs = torch.quantile( t32.abs(), INT8_CLIP_Q, dim=1) scale = (clip_abs / 127.0).clamp_min(1.0/127.0) q = torch.round(clipped / scale[:, None]) return q.to(torch.int8), scale

INT8_CLIP_Q uses quantile-based clipping to remove extreme outliers, boosting precision for the remaining values. We use the 0.9999 quantile in this project.

▶ 3

Small Tensors Don't Get Quantized

fp16 passthrough

We don't quantize every tensor. For tensors with few elements, quantization error is relatively large, and the overhead of int8 + scale actually hurts. We just cast them to fp16 and save them as-is.

Decision threshold: Tensors with INT8_KEEP_FLOAT_MAX_NUMEL or fewer elements stay in fp16 without quantization. This includes scalar parameters like attn_scale and mlp_scale.
▶ 4

zlib Compression and Roundtrip Verification

Final size reduction

After int8 quantization, we apply zlib compression on top. zlib finds repeating patterns and compresses them losslessly. The final filename extension .int8.ptz indicates this quantized+compressed format.

quant_obj, quant_stats = quantize_state_dict_int8( base_model.state_dict()) quant_buf = io.BytesIO() torch.save(quant_obj, quant_buf) quant_blob = zlib.compress(quant_buf.getvalue(), level=9) # Roundtrip verification: decompress → restore → re-evaluate quant_state = torch.load( io.BytesIO(zlib.decompress(quant_blob_disk))) base_model.load_state_dict( dequantize_state_dict_int8(quant_state))
Why roundtrip verification matters: Quantization loses information. We need to check that when we decompress and restore the model, performance hasn't degraded too much. So we remeasure val_loss and val_bpb after roundtrip.

Check Your Understanding

Why does this project use per-row scale?

Each row of a 2D matrix corresponds to a different output channel and can have different value distributions. A single scale factor would scale to the largest value in the entire matrix, losing precision in smaller values. Per-row scale lets each row be scaled to its own range, keeping precision uniform across all rows.
Chapter 10 Evaluation

bits-per-byte: A Fair Evaluation Metric

How do we measure how well a model has learned? Cross-entropy loss alone can't compare models using different tokenizers. We use bits-per-byte, a tokenizer-independent metric.

▶ 1

Why Loss Alone Isn't Enough

The tokenizer dependency problem

Cross-entropy loss measures "average information loss per token." But the same text generates many more tokens with a vocab=1024 tokenizer than with vocab=100K. More tokens make individual predictions easier, so loss looks lower—but that's a property of the tokenizer being fine-grained, not the model being better.

The key to fair comparison: We need to separate model performance from tokenizer design. bits-per-byte measures information efficiency at the byte level of the original text, so we can compare fairly across different tokenizers.
▶ 2

How bits-per-byte Works

nats → bits → bytes

bits-per-byte has three steps. First we convert cross-entropy loss from nats to bits, then from bits per token to bits per byte.

# 1. Sum loss across tokens loss_sum += per_token_loss.sum() # 2. Sum the bytes each token represents tok_bytes = base_bytes_lut[target_ids] tok_bytes += has_leading_space_lut[target_ids] \ & ~is_boundary_token_lut[prev_ids] byte_sum += tok_bytes.sum() # 3. bits-per-byte = total_loss_in_bits / total_bytes val_bpb = (loss_sum / ln(2)) / byte_sum

base_bytes_lut is the lookup table we built in Chapter 2. It records how many bytes each token represents in the original text. Whitespace handling is careful too. We accurately account for the single leading space byte represented by SentencePiece's ▁ symbol.

▶ 3

What bits-per-byte Actually Means

A measure of compression efficiency

If bits-per-byte is 1.0, the model could compress the original text to 1/8 of its size. Normally a byte needs 8 bits, but here only 1 bit would be needed. The lower the value, the better the model "understands" the text.

bits-per-byteMeaning
8.0No predictions. Pure random guessing
~5.0Basic character frequency level predictions
~1.0Very good language model
< 0.8State-of-the-art large model level

Check Your Understanding

Model A with vocab=1024 and Model B with vocab=32000 both record the same cross-entropy loss. Which model will have a lower bits-per-byte?

Even with the same loss, Model B's tokens each represent more bytes. Since bits-per-byte is (loss_in_bits * tokens) / total_bytes, Model B covers more bytes per token, giving it lower bits-per-byte. This is exactly why we can't rely on loss alone for comparison.
Chapter 11 Advanced

Test-Time Training: Per-Document LoRA Adaptation

Normally a model's weights are fixed once training ends, and we use them as-is during evaluation. But we go further: we fine-tune the model for each document during evaluation. We call this technique Test-Time Training (TTT), and we implement it efficiently with LoRA.

▶ 1

LoRA: Adapting with Few Parameters

Low-Rank Adaptation

Low-Rank Adaptation (LoRA) keeps the original weights frozen and adapts the model by learning only a small additional matrix. We add ΔW = BA to the original weight matrix W, where B and A are low-rank matrices. If A has shape (rank, input_dim) and B has shape (output_dim, rank), a small rank means very few extra parameters.

▶ 2

Per-Document Independent Adaptation: Batched LoRA

Separate adapter for each document

The heart of TTT in this project is training independent LoRA adapters for each document, identified by document boundaries at BOS tokens. We divide each document into chunks and adapt them sequentially.

class BatchedTTTLoRA(nn.Module): # Independent LoRA adapter for each batch element def __init__(self, bsz, model, rank): self.lm_head_lora = BatchedLinearLoRA( bsz, dim, vocab, rank) self.q_loras = nn.ModuleList() self.v_loras = nn.ModuleList() for block in model.blocks: self.q_loras.append( BatchedLinearLoRA(...)) # Q projection self.v_loras.append( BatchedLinearLoRA(...)) # V projection

Adapters are applied to the LM head and the Q and V projections of each block. We don't apply them to K—Q and V alone give us enough adaptation capacity while keeping parameter counts down.

▶ 3

Chunk-by-Chunk Learning and Evaluation

score-then-train

We divide each document into 256-token chunks. Within each chunk, we follow a "score first, then train" order. We record the loss for the current chunk in our bits-per-byte calculation, then use that same loss to update the LoRA adapter. This way, when we predict the next chunk, we can use patterns learned from the previous one.

Score-then-train: We use a chunk's loss for evaluation, then use the same data for training. Evaluation and training aren't separate—we train immediately after evaluation on the same data, which improves predictions for the next chunk. The final chunk is evaluated only, with no training. There's nothing to predict after it anyway.

Why TTT Matters: Computing the Competition Score

The final val_bpb

In this project, TTT LoRA results determine the final competition score. After training completes and the model goes through int8+zlib roundtrip, we apply TTT LoRA and measure val_bpb—that's our final submission score.

# Computing the final competition score torch._dynamo.reset() ttt_val_loss, ttt_val_bpb = eval_val_ttt_lora( args, base_model, rank, world_size, device, base_bytes_lut, has_leading_space_lut, is_boundary_token_lut)

The val_bpb with TTT applied is lower than the base model's val_bpb. That shows document-level adaptation meaningfully improves prediction performance.

Final Check

Why does TTT LoRA evaluate the final chunk without training on it?

TTT training has one purpose: improving predictions on the next chunk. After the final chunk, there's nothing left to predict, so training would be pointless. Plus, we avoid keeping the gradient graph, which saves memory.
Full pipeline summary: Text becomes integer IDs through the BPE tokenizer and gets stored in binary shards. During training, TokenStream reads the data and distributes it to GPUs. Token IDs become vectors through embedding, pass through 9 transformer blocks. Muon and Adam optimize matrix and scalar parameters respectively. After 20,000 steps or 10 minutes, training ends. We compress the model to under 16MB using int8 quantization + zlib compression. Final evaluation applies per-document LoRA TTT to the quantized model and measures bits-per-byte.
